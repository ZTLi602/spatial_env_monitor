## 1. 项目背景与 Phase 1 目标

本项目以 ESP32-S3 为核心，集成 SHT30 温湿度传感器、ST7789V SPI 彩色 LCD 和 HLK-LD2410C 24GHz FMCW 毫米波雷达，后续计划加入多模态数据融合和 TFLM 边缘推理。

Phase 1 的目标不是立即实现完整 GUI、机器学习和决策，而是先完成硬件链路的可用性确认：

1. 分别验证 SHT30、ST7789 和 LD2410C 能够独立工作；
2. 验证 ESP32-S3 与各外设的引脚、供电和通信协议配置正确；
3. 验证杜邦线连接环境下 SPI、I2C 和 UART 的基本稳定性；
4. 验证三个外设能够同时运行，不互相影响；
5. 建立可重复的硬件自检和日志观测流程；
6. 为 Phase 2 的传感器数据融合提供可靠底层数据。

Phase 1 完成后，系统应具备以下能力：

```text
SHT30 温湿度读取
        +
LD2410C 雷达帧接收和解析
        +
ST7789 基础显示和背光测试
        +
FreeRTOS 周期采集任务
        +
错误统计和串口日志输出
```

## 2. 硬件组成与固定约束

### 2.1 主要硬件

| 模块 | 型号 | 作用 |
|---|---|---|
| MCU | ESP32-S3-WROOM-1-N16R8 | 主控，16MB Flash，8MB PSRAM |
| LCD | 2.0 寸 ST7789V，240×320，SPI | 基础图形显示 |
| 温湿度传感器 | SHT30 I2C 模块 | 温度和相对湿度采集 |
| 毫米波雷达 | HLK-LD2410C，24GHz FMCW | 人体存在、运动和静止/微动目标检测 |
| 供电 | USB-C 5V/2A | 系统统一供电 |

### 2.2 硬件安全约束

本项目使用杜邦线连接，允许对已经核实用途的供电选择焊盘、跳线焊盘和排针进行必要焊接，但必须遵守以下流程：

1. 焊接前完全断电，不能带电焊接；
2. 焊前根据官方原理图或说明确认焊盘用途；
3. 焊后先进行目视检查和短路检查；
4. 不接外设时测量空载电压；
5. 确认电压正常后，再连接传感器；
6. 所有模块必须可靠共地；
7. 禁止并联外部电源；
8. SPI/I2C 线长控制在 10cm 以内；
9. 禁止使用 ESP32-S3 的 GPIO0、GPIO3、GPIO45、GPIO46 等 Strapping 引脚。

本次测试中，`IN/OUT` 供电选择焊盘已经按板卡用途核实并焊接，使 `5VIN` 可以输出 USB 供电的约 5V 电压。接入雷达前已完成：

- `5VIN` 与 `GND` 断电短路检查；
- `5VIN` 空载电压测量；
- 雷达 GND 与开发板 GND 共地确认；
- 无外部电源并联确认。

### 3.2 UART 交叉连接重点

雷达 UART 必须交叉连接：

```text
雷达 TX  -> ESP32 GPIO18（ESP32 UART RX）
雷达 RX  -> ESP32 GPIO17（ESP32 UART TX）
雷达 GND -> ESP32 GND
雷达 VCC -> ESP32 5VIN
```

代码中的宏从 ESP32 端命名，因此：

```c
#define LD2410C_PIN_TX   17
#define LD2410C_PIN_RX   18
```

这并不表示雷达 TX 接 GPIO17，而是表示 ESP32 UART 的 TX 为 GPIO17、RX 为 GPIO18。实际接线必须遵循 TX 对 RX、RX 对 TX。

## 4. 软件架构与模块划分

项目采用 ESP-IDF 原生组件化结构，禁止 Arduino 和 PlatformIO。每个外设驱动独立封装为 `.c/.h`，对外提供初始化、读取和释放接口。

```text
main/app_main.c
    ├── 启动硬件自检
    ├── 初始化 app_core
    ├── 启动采集任务
    └── 每秒输出样本和统计信息

components/sht30/
    ├── sht30_init()
    ├── sht30_read()
    └── sht30_deinit()

components/ld2410c/
    ├── ld2410c_init()
    ├── ld2410c_read()
    └── ld2410c_deinit()

components/st7789/
    ├── st7789_init()
    ├── st7789_fill_color()
    └── st7789_deinit()

components/self_test/
    └── 统一执行 LCD、SHT30 和 LD2410C 的基础自检

components/app_core/
    ├── 100ms 采集任务
    ├── 融合样本结构体
    ├── 32 条静态环形缓冲区
    ├── 自旋锁保护共享数据
    └── 运行统计信息
```

### 4.1 内存管理策略

按照项目约束，业务缓冲区采用静态分配，避免运行时堆分配造成不可控的内存碎片和失败风险。主要静态资源包括：

- UART 接收帧缓冲区；
- SHT30 读取缓冲区；
- app_core 采样环形缓冲区；
- FreeRTOS 静态任务栈和任务控制块；
- LCD 显示缓冲区。

### 4.2 共享数据保护

采集任务写入融合样本，主循环读取最新样本。由于 ESP32-S3 是多核 MCU，共享结构体可能被一个任务写入时被另一个任务读取，因此使用 `portMUX_TYPE` 和 `taskENTER_CRITICAL()` / `taskEXIT_CRITICAL()` 保护短时间的数据复制和索引更新。

该临界区的目的不是让任务长期独占传感器，而是保证：

- 结构体不会读到一半；
- 写指针和有效标志与样本数据保持一致；
- 不会出现新旧数据拼接。

临界区内没有放置 I2C、UART、日志或延时等可能阻塞的操作。

## 5. SHT30 Phase 1 验证

### 5.1 驱动实现要点

SHT30 使用 I2C 通信，测试流程为：

```text
发送单次高精度测量命令
        ↓
等待传感器完成转换
        ↓
读取 6 字节响应
        ↓
温度数据 CRC 校验
        ↓
湿度数据 CRC 校验
        ↓
原始值转换为摄氏温度和相对湿度
```

响应帧格式为：

```text
[T_MSB, T_LSB, T_CRC, RH_MSB, RH_LSB, RH_CRC]
```

温度和湿度分别使用 SHT30 数据手册规定的 CRC-8 校验。温度换算公式为：

```text
T = -45 + 175 × raw_temperature / 65535
```

湿度换算公式为：

```text
RH = 100 × raw_humidity / 65535
```

### 5.2 验证结果

测试日志中多次出现：

```text
SHT30 OK: T=30.9 C, RH=61.x %
```

连续采集统计达到：

```text
SHT30 ok=152 fail=0 consecutive=0
```

由此确认：

- I2C 引脚配置正确；
- SHT30 地址和通信时序正确；
- CRC 校验通过；
- 温湿度换算正确；
- 连续读取没有失败。

## 6. LD2410C Phase 1 验证

### 6.1 UART 配置

雷达采用 UART 透传上报模式，使用：

- UART1；
- 256000 baud；
- 8 数据位；
- 无奇偶校验；
- 1 个停止位；
- 无硬件流控。

初始化流程为：

```text
配置 UART 参数
        ↓
绑定 GPIO17/GPIO18
        ↓
安装 UART 驱动和 RX 缓冲区
        ↓
清空上电期间积压数据
        ↓
等待雷达自检完成
```

### 6.2 字节流同步与帧解析

UART 本质上是连续字节流，没有天然的消息边界。驱动不能假设每次读取都从一帧的第 0 字节开始，因此采用以下解析流程：

1. 逐字节搜索帧头 `F4 F3 F2 F1`；
2. 找到帧头后读取剩余帧体和帧尾；
3. 校验帧头；
4. 校验帧尾 `F8 F7 F6 F5`；
5. 校验数据长度；
6. 校验帧类型为 `0x02` 上报帧；
7. 校验数据区头尾标志 `0xAA` 和 `0x55`；
8. 校验目标状态范围；
9. 校验运动和静止能量范围为 `0~100`；
10. 按小端序提取距离字段。

解析结果统一返回：

```text
has_target
 target_state
 moving_distance_cm
 moving_energy
 static_distance_cm
 static_energy
```

### 6.3 雷达状态含义

| 状态 | 含义 |
|---:|---|
| 0 | 无目标 |
| 1 | 运动目标 |
| 2 | 静止/微动目标 |
| 3 | 运动目标和静止/微动目标同时存在 |

需要特别注意：LD2410C 并不是图像识别传感器，不会真正理解“这是人”或“这是桌子”。它主要根据毫米波回波的距离、幅度变化和微动特征判断是否存在目标。

### 6.4 实际摆手验证

测试环境为：雷达平放在桌面上，天线面朝上，手在雷达上方上下摆动。

观察到：

```text
state=2
```

会在手开始运动后变为：

```text
state=3
```

同时：

- 运动距离随手的上下位置变化；
- 运动能量可能达到 100；
- 静止/微动通道仍可能保持有效；
- 运动距离和静止距离不一定相同。

这种现象是合理的，因为同一只手、手臂、身体和桌面反射可能同时进入不同距离门，运动和静止/微动通道也不是严格互斥的分类器。

### 6.5 验证结果

日志统计达到：

```text
LD2410C ok=152 timeout=0 invalid=0 other=0 consecutive=0
```

这证明：

- UART 线路稳定；
- 雷达持续发送上报帧；
- 帧头同步成功；
- 协议校验成功；
- 运动和静止距离、能量字段解析成功；
- 连续运行期间没有通信超时和非法帧。

## 11. Phase 1 中遇到的关键问题与解决方法

### 问题一：PowerShell 找不到 `idf.py`

现象：

```text
idf.py : 无法将“idf.py”项识别为 cmdlet...
```

原因：当前终端没有加载 ESP-IDF 的环境变量和 Python 工具链路径。

解决：

```powershell
cmd.exe /k "call D:\esp\Espressif\frameworks\esp-idf-v5.2.7\export.bat"
```

或直接将环境加载和命令合并：

```powershell
cmd.exe /c "call D:\esp\Espressif\frameworks\esp-idf-v5.2.7\export.bat >nul && idf.py -p COM5 monitor"
```

### 问题二：UART 为什么必须交叉接线

UART 的 TX 是发送端，RX 是接收端，因此必须：

```text
设备 A TX -> 设备 B RX
设备 A RX <- 设备 B TX
```

本项目中雷达 TX 接 ESP32 GPIO18，雷达 RX 接 ESP32 GPIO17。通过实际日志验证了该映射正确。

### 问题三：不能只根据帧头解析雷达数据

UART 是连续字节流，启动时可能从帧中间开始读取。驱动先搜索 `F4 F3 F2 F1`，然后继续校验帧尾、长度、类型、标志和能量范围，避免把随机字节误判为合法数据。

### 问题四：雷达能量为 100 是否等于“100% 是人”

不是。能量值是雷达处理后的回波强度等级，不是目标类别置信度。雷达无法直接从日志判断“这是人”还是“这是桌子”，它主要根据运动和微动特征判断目标是否存在。

摆手测试中能量达到 100 是合理的，因为手距离近、运动方向正对雷达，反射变化很强。后续真实部署还需要进行无人环境、静止物体和人体静止对照测试。

### 问题五：为什么静止距离和运动距离不同

两个距离字段来自不同信号处理通道。它们可能分别选中手掌、手臂、身体或多径反射中的不同距离门，因此不必相同，也不代表一定存在两个目标。

## 13. Phase 1 的工程经验总结

### 13.1 先验证物理链路，再验证软件逻辑

嵌入式项目中，如果供电、共地、TX/RX 方向或焊盘用途没有确认，软件日志很容易被误判。实际流程应当是：

```text
断电检查
    ↓
焊盘用途核实
    ↓
短路检查
    ↓
空载电压检查
    ↓
连接模块
    ↓
单模块测试
    ↓
联合运行
```

### 13.2 用自检程序降低调试复杂度

把 LCD、SHT30、LD2410C 的测试集中到 `self_test` 组件中，可以将“硬件是否可用”从主业务逻辑中分离出来。这样出现问题时，可以根据日志快速判断是：

- 供电问题；
- 接线问题；
- 初始化问题；
- 协议解析问题；
- 业务任务问题。

### 13.3 对串行协议必须考虑失步

UART 不提供消息边界，不能简单地按照固定数组下标读取。可靠解析器至少需要：

- 帧头同步；
- 长度检查；
- 帧尾检查；
- 类型检查；
- 字段范围检查；
- 超时控制。

### 13.4 杜邦线环境要优先考虑信号完整性

杜邦线方便但抗干扰和阻抗控制较差。SPI/I2C 需要限制线长、保证共地、适当降低速度或驱动能力。LCD 使用半双工 SPI，并通过实际五色显示和长时间运行进行验证，是比只看“初始化成功”更可靠的验收方式。

### 13.5 日志不只是调试输出，也是验收证据

本项目使用成功次数、失败次数、超时次数、非法帧次数和连续失败次数构成运行指标。面试时可以说明：

> 我没有只通过“能打印一次数据”判断驱动正常，而是增加了可观测性指标，通过连续成功帧数、错误分类和连续失败计数验证系统稳定性。

## 14. 面试表达建议

### 14.1 项目一句话介绍

> 我使用 ESP32-S3 和 ESP-IDF 开发了一个空间感知型环境监测终端，集成 SHT30 温湿度、ST7789 SPI 显示和 LD2410C 24GHz 毫米波雷达，Phase 1 重点完成了硬件链路验证、协议解析和三模块联合稳定运行。

### 14.2 雷达驱动怎么实现

> LD2410C 通过 UART 持续输出透传帧。由于 UART 是无消息边界的字节流，我没有直接假设每次读取从帧头开始，而是先逐字节搜索 `F4 F3 F2 F1`，再读取完整帧，并依次校验帧尾、数据长度、帧类型、数据区标志、目标状态和能量范围，最后按小端序提取移动距离、静止距离和能量。

### 14.3 如何处理多核数据竞争

> 采集任务写入融合样本，主循环读取最新样本。ESP32-S3 是多核芯片，所以我使用 `portMUX_TYPE` 自旋锁保护结构体复制、写索引和有效标志更新，临界区只包含短时间内存操作，不放置日志、I2C、UART 或延时，避免阻塞系统。

### 14.4 如何保证硬件安全

> 由于项目以杜邦线和必要焊接为主，我把硬件安全作为验证流程的一部分：焊接必须完全断电，焊前核对焊盘用途，焊后做短路检查和空载电压测量，确认 5VIN 约 5V、所有模块共地后再接入雷达，且不并联外部电源。

### 14.5 如何验证不是一次性跑通

> 我增加了硬件自检和运行统计。测试不仅看单次读取是否成功，还观察连续帧成功数、UART 超时、非法帧、其他错误和连续失败次数。最终 SHT30 和 LD2410C 的成功次数持续增加，错误计数保持为零，LCD 通过五色和背光肉眼确认，并完成了长时间稳定性测试。

### 14.6 当前项目的技术边界

> LD2410C 能提供距离、运动能量和静止能量，但它不是人体分类器，不能仅凭能量值断言目标一定是人。人体和障碍物的进一步区分需要结合环境标定、规则策略或后续 TFLM 多模态模型完成。

## 15. Phase 2 衔接计划

Phase 1 已完成后，下一阶段进入数据融合：

1. 保持 Task1 以 100ms 周期采集 SHT30 和 LD2410C；
2. 统一构造六维原始输入：

```text
[温度,
 湿度,
 运动目标距离,
 静止目标距离,
 运动目标能量,
 静止目标能量]
```

3. 处理无目标时距离和能量字段的有效性；
4. 按要求归一化：
   - 温度 `[-40, 80] -> [0, 1]`；
   - 湿度 `[0, 100] -> [0, 1]`；
   - 距离 `[0, 600] -> [0, 1]`；
   - 能量 `[0, 100] -> [0, 1]`；
5. 继续使用静态环形缓冲区保存采样历史；
6. 为后续 Task2 的 TFLM 推理和决策输出准备输入接口。

Phase 2 开始前，不应破坏已经验证通过的引脚定义、供电约束、UART 帧校验和静态内存策略。
